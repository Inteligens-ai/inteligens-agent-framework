# Scrum Guide — Inteligens Agents Framework

This guide explains how the Inteligens Agents Framework implements Scrum methodology and how it integrates with your development workflow.

---

## 🎯 Overview

The framework uses **Scrum** as its primary project management methodology. The execution flow is organized around **sprints**, with specialized agents taking on Scrum roles:

- **Product Owner** - Defines backlog and user stories
- **Scrum Master** - Plans sprints and organizes work
- **Engineering Team** - Builds, tests, and delivers
- **Sprint Reviewer** - Reviews sprint outcomes
- **Sprint Closer** - Closes the sprint and prepares for next

---

## 📋 Scrum Artifacts

### 1. Product Backlog

**Generated by:** Product Owner agent (Step 1)

**Contains:**
- User stories with acceptance criteria
- Priority and estimates
- Dependencies and risks
- Product vision and scope

**Location:** `BACKLOG.md` in project root

**Example structure:**
```markdown
## Product Vision
...

## User Stories
### US-001: Feature Name
**As a** [user type]
**I want** [goal]
**So that** [benefit]

**Acceptance Criteria:**
- [ ] Criterion 1
- [ ] Criterion 2
```

### 2. Sprint Backlog

**Generated by:** Scrum Master agent (Step 2)

**Contains:**
- Sprint goal
- Selected user stories for the sprint
- Story ownership and assignments
- Sprint timeline and dependencies
- Definition of Done (sprint-level and technical)

**Location:** `SPRINT_PLAN.md` in project root

### 3. Sprint Board

**Generated by:** Scrum Master agent (Step 2)

**Visual representation of sprint progress:**
- To Do
- In Progress
- Done

**Location:** `SPRINT_BOARD.md` in project root

## 🏃 Sprint Execution Flow

### Phase 1: Planning (Steps 1-2)

**Step 1: Product Owner**
- Creates product backlog
- Defines user stories
- Sets priorities and estimates

**Step 2: Scrum Master**
- Plans sprint
- Creates sprint backlog
- Assigns stories to team members
- Defines sprint goal

**Outputs:**
- `BACKLOG.md`
- `SPRINT_PLAN.md`
- `SPRINT_BOARD.md`

### Phase 2: Design (Steps 3-4)

**Step 3: Staff Architect** ⚠️ *Requires Approval*
- Designs system architecture
- Makes architectural decisions
- Creates ADRs (Architecture Decision Records)

**Step 4: Tech Lead**
- Creates implementation plan
- Defines coding standards
- Sets up project structure

**Outputs:**
- `docs/ARCHITECTURE.md`
- `docs/ADR-*.md`
- `docs/IMPLEMENTATION_PLAN.md`

### Phase 3: Build (Steps 5-7)

**Step 5: Backend Engineer**
- Implements backend features
- Creates API endpoints
- Sets up dependencies

**Step 6: Frontend Engineer** (optional, can be skipped)
- Implements frontend features
- Creates UI components

**Step 7: AI Engineer** (optional, can be skipped)
- Implements AI/ML features
- Sets up model pipelines

**Outputs:**
- Source code
- `requirements.txt` / `package.json` / etc.
- Configuration files

### Phase 4: Test (Steps 8-10)

**Step 8: DevOps/SRE**
- Sets up CI/CD
- Creates deployment configs
- Configures monitoring

**Step 9: QA Strategist**
- Creates test plans
- Writes test cases
- Validates test coverage

**Step 10: AppSec Engineer** ⚠️ *Requires Approval*
- Security review
- Vulnerability assessment
- Security recommendations

**Outputs:**
- Test suites
- CI/CD configs
- Security reports

### Phase 5: Release (Step 11)

**Step 11: Release Manager** ⚠️ *Requires Approval*
- Prepares release
- Creates release notes
- Manages versioning

**Outputs:**
- Release documentation
- Version tags

### Phase 6: Review & Closure (Steps 12-13)

**Step 12: Sprint Reviewer**
- Reviews sprint outcomes
- Validates Definition of Done
- Identifies improvements

**Step 13: Sprint Closer** ⚠️ *Requires Approval*
- Closes sprint
- Archives sprint artifacts
- Prepares retrospective

**Outputs:**
- Sprint review report
- Retrospective notes

## 🎯 Sprint-Aware Execution

### Context Injection

During **build phases** (Steps 5-7), the Execution Runner automatically injects sprint context:

- **Current sprint goals** - What the sprint is trying to achieve
- **Focused backlog items** - Only stories in the current sprint
- **Sprint timeline** - Deadlines and milestones
- **Guardrails** - Constraints and DoD requirements

**Agents must work only within the active sprint scope.**

### Example Context Injection

When a Backend Engineer agent receives a task, it sees:

```
Sprint Context:
- Sprint Goal: Build REST API with health endpoints
- Active Stories: US-001, US-002, US-003
- Definition of Done: All tests pass, coverage ≥80%, API documented
- Timeline: 2 weeks

Focus: Implement only the stories in the sprint backlog.
```

## ✅ Definition of Done (DoD)

The framework enforces two levels of DoD:

### Sprint-Level DoD

Defined by Scrum Master in `SPRINT_PLAN.md`:
- All user stories completed
- All acceptance criteria met
- Code reviewed and merged
- Tests passing

### Technical DoD

Defined by Tech Lead in `IMPLEMENTATION_PLAN.md`:
- Code coverage ≥ 80%
- Linters passing
- Security scan clean
- Documentation complete

**Both must be satisfied before sprint closure.**

## 🔄 Sprint Lifecycle

### 1. Sprint Planning

- Product Owner presents backlog
- Scrum Master selects stories for sprint
- Team estimates and commits
- Sprint goal defined

**Framework equivalent:** Steps 1-2 (Product Owner + Scrum Master)

### 2. Daily Standups

**Not automated by framework** - Conduct manually or with your team.

**Suggested format:**
- What did I complete?
- What am I working on?
- Any blockers?

### 3. Sprint Execution

- Team works on sprint backlog items
- Continuous integration and testing
- Regular commits and reviews

**Framework equivalent:** Steps 3-11 (Design → Build → Test → Release)

### 4. Sprint Review

- Demo completed work
- Review against sprint goal
- Gather feedback

**Framework equivalent:** Step 12 (Sprint Reviewer)

### 5. Sprint Retrospective

- What went well?
- What could be improved?
- Action items for next sprint

**Framework equivalent:** Step 13 (Sprint Closer)

## 🚦 Approval Gates in Scrum Context

Certain steps require explicit approval before proceeding:

### Architecture Approval (Step 3)

**Why:** Architectural decisions have long-term impact.

**Who approves:** Technical lead or architect

**What to review:**
- Architecture decisions
- Technology choices
- System design
- ADRs

### Security Approval (Step 10)

**Why:** Security issues must be addressed before release.

**Who approves:** Security team or lead

**What to review:**
- Security scan results
- Vulnerability reports
- Security recommendations
- Compliance status

### Release Approval (Step 11)

**Why:** Releases affect production systems.

**Who approves:** Release manager or product owner

**What to review:**
- Release notes
- Version compatibility
- Deployment plan
- Rollback strategy

### Sprint Closure Approval (Step 13)

**Why:** Ensures sprint is properly closed and documented.

**Who approves:** Scrum Master or product owner

**What to review:**
- Sprint outcomes
- DoD compliance
- Retrospective notes
- Next sprint preparation

## 💡 Best Practices

### 1. Keep Sprint Scope Tight

- Focus on 3-5 user stories per sprint
- Ensure stories are independent
- Avoid over-commitment

### 2. Review Artifacts Regularly

- Check `BACKLOG.md` for product direction
- Review `SPRINT_PLAN.md` for sprint goals
- Monitor `SPRINT_BOARD.md` for progress

### 3. Validate DoD Early

- Don't wait until sprint end
- Check technical DoD after each step
- Fix issues as they arise

### 4. Use Approval Gates Wisely

- Review architecture decisions carefully
- Don't skip security reviews
- Validate releases before deployment

### 5. Document Decisions

- ADRs for architectural decisions
- Update backlog with learnings
- Keep sprint artifacts for reference
