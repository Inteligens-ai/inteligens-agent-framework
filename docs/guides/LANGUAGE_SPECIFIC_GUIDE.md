# Language-Specific Guide — Inteligens Agents Framework

This guide covers language-specific setup, dependency management, and best practices for projects generated by the framework.

> **Note:** The framework is language-agnostic. Agents will generate appropriate project structures and dependencies based on the target language. This guide helps you understand what to expect and how to work with generated projects.

---

## 🐍 Python

### Project Structure

Python projects generated by the framework typically include:

- `requirements.txt` - Python dependencies
- `setup.py` or `pyproject.toml` - Package configuration (for libraries)
- `.gitignore` - Excludes `venv/`, `__pycache__/`, `.pyc` files
- `README.md` - Project-specific documentation

### Virtual Environment Setup

**Why use a virtual environment?**

Virtual environments isolate project dependencies, preventing conflicts between different projects.

**Creating and activating:**

```bash
# Create virtual environment
python3 -m venv venv

# Activate (Linux/macOS)
source venv/bin/activate

# Activate (Windows)
venv\Scripts\activate

# Activate (PowerShell)
venv\Scripts\Activate.ps1
```

**Deactivating:**

```bash
deactivate
```

### Dependency Management

**Installing dependencies:**

```bash
pip install -r requirements.txt
```

**Note:** During framework execution, agents may update `requirements.txt` (e.g., when correcting missing dependencies). If `requirements.txt` is modified, reinstall:

```bash
pip install -r requirements.txt
```

**Common Python frameworks:**

- **FastAPI** - Modern async web framework
- **Flask** - Lightweight web framework
- **Django** - Full-featured web framework
- **pytest** - Testing framework
- **uvicorn** - ASGI server (for FastAPI)

### Running Python Projects

**Web applications (FastAPI/Flask):**

```bash
# FastAPI
uvicorn main:app --reload

# Flask
flask run
# or
python app.py
```

**Testing:**

```bash
pytest
pytest --cov=module_name --cov-report=html
```

## 📦 Node.js / TypeScript

### Project Structure

Node.js/TypeScript projects typically include:

- `package.json` - Dependencies and scripts
- `tsconfig.json` - TypeScript configuration (for TS projects)
- `.gitignore` - Excludes `node_modules/`, `.env`, `dist/`
- `README.md` - Project-specific documentation

### Dependency Management

**Installing dependencies:**

```bash
npm install
# or
yarn install
# or
pnpm install
```

**Note:** Agents may update `package.json` during execution. Reinstall dependencies after updates:

```bash
npm install
```

**Common Node.js frameworks:**

- **Express** - Web application framework
- **Fastify** - Fast web framework
- **NestJS** - Enterprise Node.js framework
- **Next.js** - React framework (full-stack)
- **Jest** - Testing framework
- **TypeScript** - Typed JavaScript

### Running Node.js Projects

**Development:**

```bash
npm run dev
# or
npm start
```

**Production:**

```bash
npm run build
npm run start:prod
```

**Testing:**

```bash
npm test
npm run test:coverage
```

## 🚀 Go

### Project Structure

Go projects typically include:

- `go.mod` - Go module definition
- `go.sum` - Dependency checksums
- `.gitignore` - Excludes `bin/`, `vendor/`
- `README.md` - Project-specific documentation

### Module Management

**Initializing a module:**

```bash
go mod init github.com/your-org/your-project
```

**Installing dependencies:**

```bash
go mod download
go mod tidy  # Remove unused dependencies
```

**Note:** Agents may update `go.mod` during execution. Run:

```bash
go mod tidy
```

**Common Go frameworks:**

- **Gin** - HTTP web framework
- **Echo** - High performance web framework
- **Fiber** - Express-inspired framework
- **Gorilla/mux** - URL router
- **Testify** - Testing toolkit

### Running Go Projects

**Development:**

```bash
go run main.go
```

**Building:**

```bash
go build -o app
./app
```

**Testing:**

```bash
go test
go test -v -cover
```

## 🦀 Rust

### Project Structure

Rust projects typically include:

- `Cargo.toml` - Project metadata and dependencies
- `Cargo.lock` - Dependency lock file (for binaries)
- `.gitignore` - Excludes `target/`
- `README.md` - Project-specific documentation

### Dependency Management

**Installing dependencies:**

Rust uses `cargo` which automatically manages dependencies:

```bash
cargo build
```

**Note:** Agents may update `Cargo.toml` during execution. Rebuild:

```bash
cargo build
```

**Common Rust frameworks:**

- **Actix-web** - Powerful web framework
- **Rocket** - Web framework with code generation
- **Warp** - Lightweight web framework
- **Tokio** - Async runtime
- **Serde** - Serialization framework

### Running Rust Projects

**Development:**

```bash
cargo run
```

**Release build:**

```bash
cargo build --release
./target/release/your-app
```

**Testing:**

```bash
cargo test
cargo test -- --nocapture  # Show output
```

## ☕ Java

### Project Structure

Java projects typically include:

- `pom.xml` (Maven) or `build.gradle` (Gradle) - Build configuration
- `.gitignore` - Excludes `target/`, `build/`, `.class` files
- `README.md` - Project-specific documentation

### Dependency Management

**Maven:**

```bash
mvn clean install
mvn dependency:resolve
```

**Gradle:**

```bash
./gradlew build
./gradlew dependencies
```

**Note:** Agents may update build files during execution. Rebuild:

```bash
# Maven
mvn clean install

# Gradle
./gradlew clean build
```

**Common Java frameworks:**

- **Spring Boot** - Enterprise framework
- **Quarkus** - Kubernetes-native framework
- **Micronaut** - Modern JVM framework
- **JUnit** - Testing framework
- **Mockito** - Mocking framework

### Running Java Projects

**Maven:**

```bash
mvn spring-boot:run
# or
java -jar target/your-app.jar
```

**Gradle:**

```bash
./gradlew bootRun
# or
java -jar build/libs/your-app.jar
```

**Testing:**

```bash
# Maven
mvn test

# Gradle
./gradlew test
```

## 🔷 C# (.NET)

### Project Structure

C# projects typically include:

- `*.csproj` or `Directory.Build.props` - Project file with dependencies
- `Program.cs` or `Startup.cs` - Application entry point
- `.gitignore` - Excludes `bin/`, `obj/`, `*.user`, `*.suo`
- `README.md` - Project-specific documentation

### Dependency Management

**Using .NET CLI:**

```bash
dotnet restore
dotnet add package PackageName
```

**Using NuGet Package Manager:**

```bash
nuget restore
```

**Note:** Agents may update `.csproj` files during execution. Restore:

```bash
dotnet restore
```

**Common C# frameworks:**

- **ASP.NET Core** - Web framework
- **Entity Framework Core** - ORM
- **xUnit** - Testing framework
- **NUnit** - Alternative testing framework
- **Moq** - Mocking framework
- **Serilog** - Logging framework

### Running C# Projects

**Development:**

```bash
dotnet run
```

**Building:**

```bash
dotnet build
dotnet build --configuration Release
```

**Running specific project:**

```bash
dotnet run --project MyProject.csproj
```

**Testing:**

```bash
dotnet test
dotnet test --collect:"XPlat Code Coverage"
```

**Publishing:**

```bash
dotnet publish -c Release
```

### .NET Version Management

**Check installed versions:**

```bash
dotnet --list-sdks
dotnet --list-runtimes
```

**Specify version in project:**

```xml
<PropertyGroup>
  <TargetFramework>net8.0</TargetFramework>
</PropertyGroup>
```

## 🔧 General Best Practices

### 1. Always Use Dependency Isolation

- **Python:** Use `venv` or `virtualenv`
- **Node.js:** `node_modules` is project-local (already isolated)
- **Go:** Modules are project-local
- **Rust:** `cargo` manages dependencies per project
- **Java:** Maven/Gradle handle dependency isolation
- **C#:** `.NET` projects use project-local `packages/` or global NuGet cache

### 2. Version Control

Always commit dependency files:
- `requirements.txt` (Python)
- `package.json` and `package-lock.json` (Node.js)
- `go.mod` and `go.sum` (Go)
- `Cargo.toml` and `Cargo.lock` (Rust)
- `pom.xml` or `build.gradle` (Java)
- `*.csproj` and `packages.lock.json` (C#)

**Never commit:**
- Virtual environments (`venv/`, `node_modules/`)
- Build artifacts (`target/`, `dist/`, `build/`)
- IDE-specific files (unless team-wide)

### 3. Dependency Updates During Execution

Agents may update dependency files when:
- Missing dependencies are detected
- Tests fail due to missing packages
- Architecture decisions require new libraries

**Always reinstall dependencies after agent updates:**

```bash
# Python
pip install -r requirements.txt

# Node.js
npm install

# Go
go mod tidy

# Rust
cargo build

# Java (Maven)
mvn clean install

# C# (.NET)
dotnet restore
```

### 4. Testing

Each language has its testing framework. Agents will generate appropriate test files:
- **Python:** `pytest`, `unittest`
- **Node.js:** `Jest`, `Mocha`
- **Go:** Built-in `testing` package
- **Rust:** Built-in `#[test]` attributes
- **Java:** `JUnit`, `TestNG`
- **C#:** `xUnit`, `NUnit`, `MSTest`

## 🎯 Framework Behavior

The framework's agents are language-aware and will:

1. **Generate appropriate project structure** for the target language
2. **Include standard dependency files** (`requirements.txt`, `package.json`, etc.)
3. **Set up testing frameworks** appropriate for the language
4. **Create language-specific `.gitignore`** files
5. **Generate language-appropriate documentation** in README files

**You don't need to specify the language** — agents infer it from the project context and requirements.
